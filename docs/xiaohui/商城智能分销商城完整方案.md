### 19.2 公海产品销售全链路

#### 19.2.1 供应商铺货到公海
```
供应商在供应商工作台发布商品
  → 设置：拿货价、库存、是否参与分销
  → 提交平台审核（若需要资质）
  → 审核通过 → 商品进入"公海商品池"（状态: pool）
  → 公海可见：所有商户 + 所有分销商
```

**商品进入公海后携带的信息：**
| 字段 | 来源 | 说明 |
|------|------|------|
| supplier_id | 供应商工作台 | 谁供货 |
| cost_price | 供应商设定 | 拿货价（平台分账用） |
| retail_price | 供应商设定（或平台建议） | 建议零售价 |
| pool_status | 审核状态 | pending / approved / rejected |
| commission_rate | 供应商可选设置 | 商品级分销佣金比例（覆盖全局） |

#### 19.2.2 商户从公海选品上架
```
商户进入"公海选品"
  → 浏览公海商品（带拿货价展示）
  → 点击"代销上架"
  → 设置本店铺零售价（必须 ≥ 拿货价，系统校验）
  → 确认代销条款（抽成比例/结算周期）
  → 上架到本店铺（状态: on_sale）

代销商品 vs 自营商品：
┌─────────────┬──────────────┬──────────────┐
│            │ 代销商品       │ 自营商品       │
├─────────────┼──────────────┼──────────────┤
│ 拿货价     │ 供应商设定     │ 商户自己定     │
│ 零售价     │ 商户设定(≥拿货价)│ 商户自己定     │
│ 差价利润   │ 零售价-拿货价  │ 零售价-成本价  │
│ 发货方     │ 供应商发货     │ 商户发货       │
│ 供应商结算 │ 周结（拿货价） │ 无             │
│ 商户结算   │ 周结（差价利润）│ 周结（利润）   │
└─────────────┴──────────────┴──────────────┘
```

**商户上架时系统自动写入 `merchant_products` 表：**
```json
{
  "merchant_id": "商户ID",
  "product_id": "公海商品ID",
  "source": "pool",           // 来源：pool=公海代销，self=自营
  "cost_price": 60,           // 拿货价（锁定，不随公海商品变动）
  "retail_price": 89,         // 本店零售价
  "status": "on_sale",
  "commission_rate": 0.375,   // 本商品分销佣金比例（默认37.5%）
  "locked_at": "2026-05-14"  // 价格锁定时间
}
```

> **价格锁定规则**：merchant_products.cost_price 和 retail_price 在上架时锁定，不随公海商品后续价格调整而自动变动，避免供应商调价影响历史订单。

### 19.3 订单分账时机与计算 ✅（V4.0已实现）

#### 19.3.1 分账触发时机

```
订单状态变更 → 触发分账判断
    │
    ├─ 状态 = completed（买家确认收货）
    │     → 立即触发自动分账
    │
    ├─ 状态 = auto_complete（超时自动确认，14天）
    │     → 立即触发自动分账
    │
    ├─ 状态 = refunded（退款完成）
    │     → 触发退款分账回滚（见 19.6）
    │
    └─ 状态 = cancelled（支付后取消，未发货）
          → 触发已支付金额退款，不走分账
```

**实现要求：** 在 `mall_orders` 状态变更处（`completed` 回调）调用 `OrderSplitService.createSplit(orderId)`，幂等校验通过后才执行分账。

#### 19.3.2 分账计算公式

**示例数据：**
- 零售价：¥89
- 拿货价：¥59（供应商供货价）
- 平台服务费率：3%
- 供应商补贴分销佣金比例：37.5% of 毛利（L1）

**第一步：计算基础金额**
```
商品毛利 = 零售价 - 拿货价 = 89 - 59 = ¥30
订单总额 = ¥89 × 数量（假设1件）
```

**第二步：计算分账各方**
```
平台服务费 = 订单总额 × 3% = 89 × 3% = ¥2.67（四舍五入¥2.7）
供应商实收 = 拿货价 × 数量 = ¥59

差价利润 = 订单总额 - 平台服务费 - 供应商实收
         = 89 - 2.7 - 59 = ¥27.3
         （也等于 毛利 - 平台服务费 = 30 - 2.7）

分销佣金L1 = 差价利润 × 37.5% = 27.3 × 0.375 = ¥10.24
分销佣金L2 = 差价利润 × 12.5% = 27.3 × 0.125 = ¥3.41
商户净收 = 差价利润 - 分销佣金L1 - 分销佣金L2
        = 27.3 - 10.24 - 3.41 = ¥13.65
```

**验证：¥2.7 + ¥59 + ¥13.65 + ¥10.24 + ¥3.41 = ¥89 ✅**

**分账汇总表：**
| 接收方 | 金额 | 计算公式 | 到账时间 |
|--------|------|----------|----------|
| 平台 | ¥2.70 | 订单总额 × 3% | 即时 |
| 供应商 | ¥59.00 | 拿货价 × 数量 | 周结（每周一） |
| 商户 | ¥13.65 | 差价利润 - 分佣L1 - 分佣L2 | 周结（每周一） |
| 分销商L1 | ¥10.24 | 差价利润 × 37.5% | 冻结7天后可提现 |
| 分销商L2 | ¥3.41 | 差价利润 × 12.5% | 冻结7天后可提现 |

### 19.4 记账系统（Account Ledger）✅（V4.0已实现）

#### 19.4.1 账户结构

每个参与者（供应商/商户/分销商）在平台上有一个**资金账户**：

| 账户 | 所有者 | 资金来源 | 体现方式 |
|------|--------|----------|----------|
| 供应商账户 | supplier | 买家确认收货后，订单拿货价进入 | 周结后可用余额 |
| 商户账户 | merchant | 买家确认收货后，差价利润进入 | 周结后可用余额 |
| 分销商账户 | distributor | 佣金冻结7天后解冻进入 | 可提现余额 |
| 平台账户 | platform | 服务费即时+佣金抽成 | 平台自有资金 |

**分销商账户的特殊性：**
```
分销商账户余额 = 冻结金额 + 可提现金额
  冻结金额：订单完成，佣金计算后进入冻结（防退款）
  可提现金额：冻结7天届满，自动转入
```

#### 19.4.2 账户变动流水（`account_ledger` 表）

**核心原则：一笔订单 → 多笔分账 → 每笔分账对应一条 ledger 记录**

```json
// 订单完成，写入 5 条 ledger 记录（各方各一条）
{
  "id": "ledger_xxx001",
  "account_type": "supplier",
  "account_id": "供应商ID",
  "order_id": "订单ID",
  "split_item_id": "item_002",
  "amount": 59.00,
  "direction": "credit",        // credit=入账，debit=出账
  "balance_before": 1000.00,
  "balance_after": 1059.00,
  "transaction_type": "order_income",
  "note": "订单#20260514001拿货价收入",
  "created_at": "2026-05-14 10:00:00"
}

{
  "id": "ledger_xxx002",
  "account_type": "merchant",
  "account_id": "商户ID",
  "order_id": "订单ID",
  "split_item_id": "item_003",
  "amount": 13.65,
  "direction": "credit",
  "balance_before": 500.00,
  "balance_after": 513.65,
  "transaction_type": "order_income",
  "note": "订单#20260514001差价利润",
  "created_at": "2026-05-14 10:00:00"
}

{
  "id": "ledger_xxx003",
  "account_type": "distributor",
  "account_id": "分销商L1ID",
  "order_id": "订单ID",

  "amount": 10.24,
  "direction": "credit",
  "balance_before": 0,
  "balance_after": 0,           // 仍在冻结中，不计入可提现
  "transaction_type": "commission_frozen",  // 冻结中

  "note": "订单#20260514001佣金（冻结至5月21日）",
  "created_at": "2026-05-14 10:00:00"
}
```

#### 19.4.3 账户余额计算

**账户余额永远来自 ledger 的 sum 汇总，而非单独存储：**

```sql
-- 查询分销商可提现余额
SELECT 
  COALESCE(SUM(CASE WHEN direction = 'credit' THEN amount ELSE 0 END), 0) -
  COALESCE(SUM(CASE WHEN direction = 'debit' THEN amount ELSE 0 END), 0) -
  COALESCE(SUM(CASE WHEN transaction_type = 'commission_frozen' THEN amount ELSE 0 END), 0)
FROM account_ledger
WHERE account_type = 'distributor' AND account_id = :distributor_id;

-- 冻结金额（佣金已解冻但未提现）
SELECT COALESCE(SUM(amount), 0)
FROM account_ledger
WHERE account_type = 'distributor' 
  AND account_id = :distributor_id
  AND transaction_type = 'commission_frozen'
  AND frozen_until <= NOW();

-- 可提现余额（解冻后自动转入）
SELECT COALESCE(SUM(amount), 0)
FROM account_ledger
WHERE account_type = 'distributor'
  AND account_id = :distributor_id
  AND transaction_type = 'commission_available';
```

### 19.5 提现申请与打款流程⚠️（payout.js已有提现申请框架，审核/打款/回调待实现）

#### 19.5.1 提现前置条件
```
1. 分销商账户有可提现余额（> 0）
2. 可提现余额 >= 最低提现门槛（默认 ¥100）
3. 当前没有正在处理中的提现申请（防止重复申请）
4. 账户未被冻结（违规处罚）
```

#### 19.5.2 提现申请流程
```
分销商点击"申请提现"
    ↓
系统计算：
  可提现余额：¥500.00
  最低提现门槛：¥100
  输入提现金额：¥300
  手 续 费：¥300 × 0.1% = ¥0.30（微信），封顶 ¥25
  实际到账：¥299.70
    ↓
分销商确认 → 填写微信收款码（首次）
    ↓
生成提现记录（状态：pending）
    ↓
平台管理员收到通知（后台可见）
    ↓
管理员审核：同意 / 拒绝
    ↓
├─ 拒绝：余额释放回可提现账户，状态=failed，附拒绝原因
└─ 同意：打款操作（手动或自动）
           ↓
        打款成功 → 状态=completed
        账户变动：可提现余额 - ¥300，ledger写debit记录
```

**提现记录状态机：**
```
pending（申请中）
  ↓ 管理员审核通过
processing（打款中）
  ↓ 打款成功
completed（已完成）
  ↓ 打款失败
failed（已失败，余额释放回账户）
  ↓ 管理员拒绝
rejected（已拒绝，余额释放回账户）
```

#### 19.5.3 提现手续费规则
| 提现方式 | 手续费率 | 封顶 |
|----------|----------|------|
| 微信零钱 | 0.1% | ¥25 |
| 银行卡 | 0.05% | ¥50 |
| 企业转账到微信 | 0.1% | ¥25 |

#### 19.5.4 自动打款 vs 手动打款
| 模式 | 说明 | 适用场景 |
|------|------|----------|
| 手动打款 | 管理员审核后手动转账 | 小额/初期 |
| 自动打款 | 审核通过后，系统调用微信付款API/支付宝转账API自动打款 | 成熟运营 |

**自动打款前提：**
- 平台开通企业付款到微信零钱能力（微信支付商户号）
- 或支付宝商家转账能力

### 19.6 退款时分账回滚✅（V4.0已实现：Saga补偿+乐观锁）

#### 19.6.1 退款分账回滚触发
```
买家申请退款
  → 卖家同意（或平台介入）
  → 退款完成（订单状态 → refunded）
  → 触发退款分账回滚
  → 按各方已分账金额原路扣回
```

#### 19.6.2 回滚计算规则

**退款金额由各方按分账比例分担：**
```
退款金额 = 买家实际支付金额（减去已使用的优惠券等）

各方分账比例（从 order_split_items 读取）：
  平台：2.70 / 89 = 3.03%
  供应商：59.00 / 89 = 66.29%
  商户：13.65 / 89 = 15.34%
  分销商L1：10.24 / 89 = 11.51%
  分销商L2：3.41 / 89 = 3.83%

各方应扣回金额：
  平台：退款金额 × 3.03%
  供应商：退款金额 × 66.29%
  商户：退款金额 × 15.34%
  分销商L1：退款金额 × 11.51%
  分销商L2：退款金额 × 3.83%
```

#### 19.6.3 回滚时机与状态

**根据分账状态决定扣回方式：**

| 分账状态 | 场景 | 处理方式 |
|----------|------|----------|
| 未结算（金额还在平台账户） | 订单完成不久，还在周结前 | 直接从待结算金额中扣除，无需实际资金转移 |
| 已结算（钱已打到供应商/商户账户） | 周结算完成后发生退款 | **需要追回**：从下期结算金额中扣除，或从账户余额中扣 |
| 佣金已提现 | 分销商已提现后发生退款 | 从可提现余额中扣除；若余额不足，标记欠款，下次佣金优先偿还 |

#### 19.6.4 退款回滚流程
```
1. 查询 order_splits，确认各方分账金额
2. 查询各方账户当前余额
3. 按比例计算各方应扣回金额
4. 写入退款回滚记录（refund_splits 表）
5. 写入各方 ledger（direction: debit）
6. 更新 order_splits.refund_status = 'rolled_back'
7. 若分销商佣金已提现但余额不足：
     → 写入 distributor_debts 表（欠款记录）
     → 后续佣金结算时优先偿还
```

### 19.9 分账安全与风控✅（Money类+乐观锁+幂等+事务，V4.0已全部实现）

#### 19.9.1 分账前置校验
```javascript
// 每次分账前，必须校验以下条件
async function beforeSplit(orderId) {
  const order = await prisma.mall_orders.findUnique({ where: { id: orderId } });
  
  // 1. 订单状态必须是 completed
  if (order.status !== 'completed') {
    throw new Error('ORDER_NOT_COMPLETED');
  }
  
  // 2. 订单未发生过退款
  if (order.refunded_amount > 0) {
    throw new Error('ORDER_ALREADY_REFUNDED');
  }
  
  // 3. 订单尚未分账（幂等性保证）
  const existing = await prisma.order_splits.findUnique({ where: { order_id: orderId } });
  if (existing) {
    throw new Error('ORDER_ALREADY_SPLIT');
  }
  
  // 4. 分账金额必须与订单金额一致
  const items = await calculateSplit(orderId);
  const total = items.reduce((sum, i) => sum + i.amount, 0);
  if (Math.abs(total - order.total_amount) > 0.01) {
    throw new Error('SPLIT_AMOUNT_MISMATCH');
  }
  
  return true;
}
```

#### 19.9.2 幂等保证
```
同一订单分账只能成功一次：
  - 分账前先查 order_splits 是否已存在
  - 存在则跳过，防止重复分账
  - 写入时用事务：先写 order_splits，再写各 order_split_items，再更新各账户 ledger
```

#### 19.9.3 事务保证
```javascript
await prisma.$transaction(async (tx) => {
  // 1. 写入分账主记录
  const split = await tx.order_splits.create({ data: splitData });

  // 4. 更新订单分账状态

  });
});
```

### 21.1 合规升级：监管分账模式（彻底规避二清）

#### 21.1.1 二清风险与合规改造背景

> 2026年金融监管环境：人民银行明确禁止无证机构从事资金清算（"二清"）。
> 原方案：平台先收买家的钱，再由平台分发给供应商/商户 → 涉嫌二清。
> V4.0改造：资金不落地，平台只发指令，由持牌机构（银行/支付公司）执行资金划拨。

#### 21.1.2 监管分账架构

```
买家付款
  ↓
资金进入银行/持牌机构监管子账户（平台不过钱）
  ↓
系统生成 settlement_order（分账指令）推送给银行/第三方支付
  ↓
银行按指令将资金直接划拨到各方子账户（T+0 或周结）
  ↓
平台只记录指令流水，不碰资金
```

**关键词：** `资金不落地` `指令代替资金` `持牌机构执行`

#### 21.1.3 第三方分账平台接入（Ping++ / 银联分账）

| 平台 | 适合场景 | 分账比例上限 | 结算周期 |
|------|----------|-------------|----------|
| 微信支付分账 | 微信内支付 | 单次最高50方 | T+0 / T+1 |
| 支付宝分账 | 支付宝内支付 | 单次最高50方 | T+0 |
| 银联商服通 | 银行卡/跨平台 | 按协议 | 可配置 |
| Ping++ 分账云 | 聚合以上能力 | 同上 | 同上 |


```
  ├─ FAILED → 更新 order_splits.status = FAILED，触发告警，人工介入
```

### 21.4 架构加固：高可用与亿级数据保障

#### 21.4.1 Redis库存预占机制（防超卖）

**场景：** 大促期间，A4打印纸限量5000件，1秒内涌入10000次下单请求。

**原方案问题：** 直接UPDATE数据库 → 锁表 → 超卖或性能崩溃。

**V4.0方案：Redis原子扣减 → 最终一致性**

```
买家下单 → 减库存
  ↓
Redis Lua脚本（原子操作）：
  local stock = redis.call('GET', 'stock:product:001')
  if tonumber(stock) < tonumber(ARGV[1]) then
    return -1  -- 库存不足
  end
  redis.call('DECRBY', 'stock:product:001', ARGV[1])
  return 1    -- 扣减成功
  ↓
支付成功 → 数据库异步扣减（乐观锁）
  ↓
支付超时（30分钟）→ Redis回滚库存（Lua脚本加回去）
```

```javascript
// 库存预占接口定义
POST /api/v1/inventory/reserve

Request:
{
  "product_id": "P001",
  "quantity": 5,
  "order_id": "mo-20260514-001",    // 关联订单
```

#### 21.4.3 订单分表策略（按月归档）

**背景：** 预计2026年底日订单量1万+，年度订单量360万+，单表无法承载。

**分表规则：**
```
mall_orders_202605  -- 2026年5月订单
mall_orders_202606  -- 2026年6月订单
...
mall_orders_202612  -- 2026年12月订单

路由逻辑：
  order_id = 'mo-20260514-001'
  → 取 order_id 第4-7位年月 = '202605'
  → 路由到 mall_orders_202605 表
```

**实现方式（Prisma Raw SQL 路由）：**
```javascript
// 分表路由辅助函数
function getOrderTableSuffix(orderId) {
  // orderId格式: mo-YYYYMMDD-xxxxx
  const match = orderId.match(/^mo-(\d{4}\d{2})\d{2}-/);
  return match ? match[1] : getCurrentMonth(); // fallback到当前月
}

async function queryOrder(orderId) {
  const suffix = getOrderTableSuffix(orderId);
  const table = `mall_orders_${suffix}`;
  return prisma.$queryRaw`SELECT * FROM ${table} WHERE id = ${orderId}`;
}
```

**分表切换时机：** 每月最后一天23:00自动创建下月空表（Crond）。
